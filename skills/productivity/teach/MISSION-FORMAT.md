# MISSION.md 格式

`MISSION.md` 位于工作区根目录，捕获用户学习这个主题的**原因**。每个教学选择都应追溯到这份文档。

## 模板

```md
# 任务：{主题}

## 为什么

1-3 句话。用户在追逐的具体现实世界目标。当他们掌握这项技能后，他们的生活或工作会有什么变化？避免抽象表述（如"理解 X"）。

示例对比：
- 好："在十月份之前跑完半程马拉松。"
- 差："变得更健康。"

- 好："在年底前为我的 Rust CLI 工具获得 100 个 GitHub Star。"
- 差："学习 Rust。"

## 成功的样子

用户将能够做的具体、可观察的事情。

- [ ] {可验证的结果}
- [ ] {可验证的结果}

## 约束条件

时间、预算、先前承诺、学习偏好，任何限定方法的东西。

## 不在范围内

用户目前明确不想追逐的相邻主题。保护最近发展区。
```

## 规则

1. **一个工作区一个任务。** 两个无关主题 = 两个工作区。
2. **具体优于抽象。** "在十月份之前跑完半程马拉松"优于"变得更健康"。"为我的 Rust CLI 工具获得 100 个 GitHub Star"优于"学习 Rust"。
3. **对模糊要追问。** 如果用户无法阐述他们的"为什么"，先访谈他们。差的任务比没有任务更坏。
4. **当现实变化时修订。** 当目标改变时更新文件。不要让过时的任务引导未来的 session。
5. **保持简短。** 如果文件超过一屏，它就不再是指南针，而是变成了计划。
