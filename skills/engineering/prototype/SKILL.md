# Prototype（原型）

原型是**回答一个问题的一次性代码**。问题决定了它的形状。

## Pick a branch（选择分支）

确定正在回答的是哪个问题——从用户提示、周围代码，或如果用户在的话直接问：

- **"这个逻辑/状态模型感觉对吗？"** → [LOGIC.md](LOGIC.md)。构建一个小的交互式终端应用，推动状态机通过那些在纸上难以推理的情况。
- **"这应该长什么样？"** → [UI.md](UI.md)。在单一路由上生成几个截然不同的 UI 变体，通过 URL 搜索参数和浮动的底部栏切换。

两个分支产生非常不同的产物——选错了会浪费整个原型。如果问题确实模糊且用户不在，默认选择更适合周围代码的分支（后端模块 → logic；页面或组件 → UI），并在原型顶部陈述你的假设。

## Rules that apply to both（两者都适用的规则）

1. **从第一天就是一次性的，并明确标记。** 将原型代码放在它实际使用位置的附近（正在为其做原型的模块或页面旁边），这样上下文很明显——但命名让它能看出是原型，而不是生产代码。对于一次性 UI 路由，遵循项目已经使用的任何路由约定；不要发明新的顶层结构。

2. **一个命令就能运行。** 不管项目的运行器是什么——`pnpm <name>`、`python <path>`、`bun <path>` 等。用户必须能不假思索地启动它。

3. **默认无持久化。** 状态存在于内存中。持久化是原型要_检查_的东西，而不是它应该依赖的东西。如果问题明确涉及数据库，使用一个临时 DB 或带有明确"PROTOTYPE — wipe me"名称的本地文件。

4. **跳过打磨。** 没有测试，没有超出让原型_可运行_的错误处理，没有抽象。目标是为了快速学习。

5. **暴露状态。** 每次动作后（logic）或每次变体切换时（UI），打印或渲染全部相关状态，让用户能看到什么发生了变化。

6. **完成后捕获它。** 将任何已验证的决策整理到真实代码中，然后将原型本身作为**一手来源**捕获：将其提交到一次性分支，远离 main，并在实现 issue 上留下指向该分支的上下文指针。也要捕获答案——结论和它解决的问题——在 issue 或 commit 中。main 分支只保留已验证的决策。
