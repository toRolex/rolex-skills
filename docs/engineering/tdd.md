## 功能

`tdd` 用测试先行的方式构建功能或修复 bug，一次一个行为，通过红-绿循环驱动代码。

它**不会**一次性写所有测试。批量写测试（水平切片）会产生测试*想象出来的*行为——它们检查代码的形状，对真实变化不敏感。`tdd` 采用垂直切片：一个测试，然后刚够通过的代码，然后下一个测试，每个循环由上一个教给你的东西驱动。测试只针对公共接口，所以下面的实现可以变化而不影响测试。

## 何时使用

敲 `/tdd` 或 agent 自动调用。

当有具体行为要构建，且想要能经受重构的测试时使用。如果行为还没确定，先用 `/to-spec`。如果工作重点是接口形状而非测试，用 `/codebase-design`。

## 一次一个切片

核心思路是**红-绿循环**：写一个失败的测试（红），加刚好够的代码让它通过（绿），然后重复下一个行为。第一个循环是**tracer bullet**：一个测试证明单个路径端到端工作，然后从那里向外构建。

## 效果良好的标志

- 一次写一个测试，让它通过，然后写下一个——不是一批测试跟一批代码
- 测试命名行为而非内部实现
- 预期值是 spec 的字面值，不是用代码相同方式推导出来的

## 在流程中的位置

`tdd` 是主构建链中 `/implement` 内部驱动的红-绿引擎。不确定时问 `/ask-rolex`。
