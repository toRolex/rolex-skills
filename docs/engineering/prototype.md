## 功能

`prototype` 构建一个小的、一次性程序，唯一的任务是回答一个设计问题——这个状态模型对吗，或者这个 UI 应该长什么样。

代码**从第一天就是一次性的**，并明确标记。没有测试，没有超出可运行性的错误处理，没有抽象，没有持久化。目标是为了快速学习然后删除。

## 何时使用

敲 `/prototype` 或 agent 自动调用。

当有一个难以在纸上确定的设计问题时使用——一个状态机有你无法在脑子里装下的情况，或者一个屏幕你需要看到几个版本并排才能确认。如果是已有功能出问题了，用 `/diagnosing-bugs`。

## 两个分支

问题决定形状：

- **"这个逻辑/状态模型对吗？"** → 一个小交互终端应用，推动状态机通过边界情况
- **"这应该长什么样？"** → 同一页面的几个截然不同的 UI 变体，方便比较

## 保留原型作为一手来源

完成的原型留下两样东西。**答案**——结论和它解决的问题——是你要持久记录的内容。**原型本身是一手来源**——答案来自的可运行证据，存在一次性分支上，不合并到 main。

## 在流程中的位置

随时可用的独立工具。答案通常喂养下一步——经过验证的状态模型或 UI 方向成为 `/to-spec` 的输入。不确定时问 `/ask-rolex`。
